## 2026-06-07 — LoopDriver Phase G — COMPLETE (per-signal isolation + IN-001 kill-switch single source)
Closed the two remaining Phase G increments; the deterministic LoopDriver is now feature-complete (A–H + the Phase G closers). **(a) §8.4 per-signal exception isolation** — each `process_signal` in `LoopDriver._dispatch_signals` is wrapped so one signal's failure cannot kill the loop: the failure is logged, journaled `BROKER_ERROR` (WARNING, §15.4; metadata `error`/`signal_id` — the signal's explicit id or a composed `strategy_id:symbol:timestamp`, not the handler's internal derivation), surfaced as an edge-triggered telemetry log line, and routing continues to the next signal/bar (Constitution §6 — logged, never swallowed). `EXECUTION_CALLS` is metered only on a successful call. New `RuntimeTelemetryPublisher.publish_log(level, message)` added to the bridge + `TelemetryTransport` Protocol (the real ZMQ transport already satisfies it; `telemetry.logs.{node}`). **(b) IN-001 `KILL_SWITCH_ACTIVATED` single source** — emission moved from the watchdog **data-health proxy** to a single observation of the handler's own kill-switch edge (`_kill_switched` False→True, §10.7) in the new `LoopDriver._check_kill_switch`, run each tick after routing and the watchdog drive. Every cause — drawdown / broker / daily-trade-limit **and** stale data — is now journaled exactly once, in replay/paper as well as live; the proxy emission (and its `KILL_SWITCH_EVENTS` meter) was removed from `_drive_watchdog`, while `WATCHDOG_STALE_DATA` stays there. Edge-triggered via a `_kill_switch_was_active` latch. The real watchdog trips the kill switch via `handler.activate_kill_switch()` (`watchdog.py:82`); the stale-data tests were rewired so `FakeWatchdog` shares the driver's handler and trips it the same way (preserving `WATCHDOG_STALE_DATA` → `KILL_SWITCH_ACTIVATED` ordering). **IN-001 is Resolved** (`ARCHITECTURE_DECISIONS.md`). TDD (RED→GREEN): +17 tests — 3 bridge `publish_log`, 8 §8.4 (`tests/runtime/test_driver_broker_error.py`), 6 IN-001 (`tests/runtime/test_driver_kill_switch.py`); full suite **234 → 251 passing**, no regressions.

## 2026-06-06 — LoopDriver Runtime Observability Layer (runtime metrics counters)
Added an internal, in-process runtime telemetry layer to the `LoopDriver`: a small, inert-by-default, dependency-injected metric-counter seam (`core/runtime/metrics.py`) that gives visibility into runtime behavior **without changing it**. `RuntimeMetric` enumerates exactly 12 in-scope counters — lifecycle (`startup`/`recovery`/`reconciliation`/`stop`), runtime (`bars_processed`/`signals_received`/`signals_routed`/`loop_iterations`), watchdog (`heartbeats_emitted`/`stale_data_events`/`kill_switch_events`), execution (`execution_calls`); no broker/PnL/portfolio metrics. `TelemetrySink` is the injected interface (`increment(metric, count=1)`); `NullTelemetrySink` is the inert default and `InMemoryTelemetrySink` accumulates counts (`get`/`snapshot`). The driver bumps counters at each existing observable point via a `_meter()` helper that swallows-and-logs any sink failure (`setup_logger("loop_driver")`), so **telemetry is best-effort and can never stop trading**; no control flow branches on telemetry state, so "telemetry never affects decisions" holds structurally. Observation only — never a source of truth, never in the trade decision path (ADR-001), no globals/singletons. `STOP_COUNT` counts only a clean `finalize_stop` (refuse-to-start is not a stop); `RECONCILIATION_COUNT`/`RECOVERY_COUNT` count every cycle (incl. vacuous/fail). **Naming:** this is the **"Runtime Observability Layer"**, recorded as a standalone milestone outside the A–H execution-build sequence — it is deliberately **distinct from** the still-planned §10 ZMQ `TelemetryPublisher` wire transport (the roadmap "Phase H — Telemetry"), and is a **foundation** that future wire publisher can read from and publish. TDD: `tests/runtime/test_telemetry_sink.py` (9) + `tests/runtime/test_driver_telemetry.py` (11); runtime suite **155 → 175 passing** (existing startup-gate / execution-routing / watchdog suites unchanged).
*Ref: docs/DRIVER_SPECIFICATION.md (observability / §10); Constitution Principle 1, §6; ADR-001/002/006; docs/PROJECT_STATE.md.*

## 2026-06-06 — LoopDriver Execution Routing — routing sub-slice (Phase G)
Wired signal routing into the `LoopDriver` (`core/runtime/driver.py` `_dispatch_signals`): each pulled `SignalEvent` is forwarded, in list order, to the canonical `ExecutionHandler.process_signal(signal, current_price=bar.close)` (§8.1). `current_price` is always `bar.close` — deterministic, so live and replay traverse the identical routing path. Routing is gated on handler presence **and** the RUNNING state (PAUSED suspends it without going blind, §3.1); the no-handler path (replay/inert/test) collects only, and `signals_pulled` still counts unconditionally (Phase D preserved). `_tick` is untouched, so watchdog `record_bar`/heartbeat ordering is byte-for-byte unchanged. ADR-006: the driver is the sole runtime caller of `process_signal`; the `SignalSource` seam holds no handler handle (statically guarded). The Phase E/F "routes nothing" AST guard inverted to `test_driver_routes_through_process_signal`; every watchdog behavioral test is unchanged. Deliberately narrow — no orders/fills/sizing/risk/telemetry. Added `tests/runtime/test_driver_execution_routing.py` (11 tests) + a `process_signal` recording spy on `FakeExecutionHandler`; runtime suite **155 passing**. Runtime flow now `Bar → Clock → SignalSource → LoopDriver → ExecutionHandler → RuntimeWatchdog → RuntimeEventJournal`. **Deferred (close Phase G next):** per-signal exception isolation (§8.4, `BROKER_ERROR`) and the IN-001 `KILL_SWITCH_ACTIVATED` single-source migration — now reachable, since `process_signal` can trip the handler's own kill switch while the driver still observes only `watchdog.data_healthy`.
*Ref: docs/DRIVER_SPECIFICATION.md §8; ADR-005, ADR-006; ARCHITECTURE_DECISIONS.md IN-001; docs/PROJECT_STATE.md.*

## 2026-06-06 — LoopDriver Startup Gate / Recovery (Phase F)
Added the §11 startup-validation gate to the `LoopDriver` (`core/runtime/driver.py`), run inside `run()` before the loop when an `ExecutionHandler` is injected. Sequence: `RECOVERY_STARTED` → reuse the handler's already-run recovery (`load_db_state=True`; the driver **never** calls `_replay_state()` — ADR-001) → `RECOVERY_COMPLETED` → reconcile the restored ledger against broker truth (`handler.reconciliation.reconcile(broker_positions)`) → empty = `RECONCILIATION_PASS` → `RUNNING`, non-empty = `RECONCILIATION_FAIL` → `abort_startup()` (STOPPED + critical alert; tick loop never runs). Reconciliation branches by **source presence** (require flag + injected `broker_positions`), not by mode; vacuously clear for paper/replay and for LIVE until the broker book is wired (Planned #6). **LIVE mode now requires an injected `ExecutionHandler`** — `run()` raises `RuntimeError` without one; the no-handler ungated path is replay/inert/test only. Still execution-free — no `process_signal` (ADR-006); the watchdog guard was narrowed from "no `ExecutionHandler` dependency" to the real ADR-006 invariant (no `process_signal` call), allowing the legitimate handler reference the gate requires. Phase E watchdog tests updated to inject a `FakeExecutionHandler`; added `FakeExecutionHandler`/`FakeReconciliation` + `test_driver_startup_gate.py`; runtime suite increased to **144 passing**.
*Ref: docs/DRIVER_SPECIFICATION.md §11; ADR-001, ADR-006; docs/PHASE_F_STARTUP_GATE_PLAN.md; docs/PROJECT_STATE.md.*
