## 2026-06-06 ‚Äî LoopDriver Runtime Observability Layer (runtime metrics counters)
Added an internal, in-process runtime telemetry layer to the `LoopDriver`: a small, inert-by-default, dependency-injected metric-counter seam (`core/runtime/metrics.py`) that gives visibility into runtime behavior **without changing it**. `RuntimeMetric` enumerates exactly 12 in-scope counters ‚Äî lifecycle (`startup`/`recovery`/`reconciliation`/`stop`), runtime (`bars_processed`/`signals_received`/`signals_routed`/`loop_iterations`), watchdog (`heartbeats_emitted`/`stale_data_events`/`kill_switch_events`), execution (`execution_calls`); no broker/PnL/portfolio metrics. `TelemetrySink` is the injected interface (`increment(metric, count=1)`); `NullTelemetrySink` is the inert default and `InMemoryTelemetrySink` accumulates counts (`get`/`snapshot`). The driver bumps counters at each existing observable point via a `_meter()` helper that swallows-and-logs any sink failure (`setup_logger("loop_driver")`), so **telemetry is best-effort and can never stop trading**; no control flow branches on telemetry state, so "telemetry never affects decisions" holds structurally. Observation only ‚Äî never a source of truth, never in the trade decision path (ADR-001), no globals/singletons. `STOP_COUNT` counts only a clean `finalize_stop` (refuse-to-start is not a stop); `RECONCILIATION_COUNT`/`RECOVERY_COUNT` count every cycle (incl. vacuous/fail). **Naming:** this is the **"Runtime Observability Layer"**, recorded as a standalone milestone outside the A‚ÄìH execution-build sequence ‚Äî it is deliberately **distinct from** the still-planned ¬ß10 ZMQ `TelemetryPublisher` wire transport (the roadmap "Phase H ‚Äî Telemetry"), and is a **foundation** that future wire publisher can read from and publish. TDD: `tests/runtime/test_telemetry_sink.py` (9) + `tests/runtime/test_driver_telemetry.py` (11); runtime suite **155 ‚Üí 175 passing** (existing startup-gate / execution-routing / watchdog suites unchanged).
