## 2026-06-06 ‚Äî LoopDriver Runtime Observability Layer (runtime metrics counters)
Added an internal, in-process runtime telemetry layer to the `LoopDriver`: a small, inert-by-default, dependency-injected metric-counter seam (`core/runtime/metrics.py`) that gives visibility into runtime behavior **without changing it**. `RuntimeMetric` enumerates exactly 12 in-scope counters ‚Äî lifecycle (`startup`/`recovery`/`reconciliation`/`stop`), runtime (`bars_processed`/`signals_received`/`signals_routed`/`loop_iterations`), watchdog (`heartbeats_emitted`/`stale_data_events`/`kill_switch_events`), execution (`execution_calls`); no broker/PnL/portfolio metrics. `TelemetrySink` is the injected interface (`increment(metric, count=1)`); `NullTelemetrySink` is the inert default and `InMemoryTelemetrySink` accumulates counts (`get`/`snapshot`). The driver bumps counters at each existing observable point via a `_meter()` helper that swallows-and-logs any sink failure (`setup_logger("loop_driver")`), so **telemetry is best-effort and can never stop trading**; no control flow branches on telemetry state, so "telemetry never affects decisions" holds structurally. Observation only ‚Äî never a source of truth, never in the trade decision path (ADR-001), no globals/singletons. `STOP_COUNT` counts only a clean `finalize_stop` (refuse-to-start is not a stop); `RECONCILIATION_COUNT`/`RECOVERY_COUNT` count every cycle (incl. vacuous/fail). **Naming:** this is the **"Runtime Observability Layer"**, recorded as a standalone milestone outside the A‚ÄìH execution-build sequence ‚Äî it is deliberately **distinct from** the still-planned ¬ß10 ZMQ `TelemetryPublisher` wire transport (the roadmap "Phase H ‚Äî Telemetry"), and is a **foundation** that future wire publisher can read from and publish. TDD: `tests/runtime/test_telemetry_sink.py` (9) + `tests/runtime/test_driver_telemetry.py` (11); runtime suite **155 ‚Üí 175 passing** (existing startup-gate / execution-routing / watchdog suites unchanged).
*Ref: docs/DRIVER_SPECIFICATION.md (observability / ¬ß10); Constitution Principle 1, ¬ß6; ADR-001/002/006; docs/PROJECT_STATE.md.*

## 2026-06-06 ‚Äî LoopDriver Execution Routing ‚Äî routing sub-slice (Phase G)
Wired signal routing into the `LoopDriver` (`core/runtime/driver.py` `_dispatch_signals`): each pulled `SignalEvent` is forwarded, in list order, to the canonical `ExecutionHandler.process_signal(signal, current_price=bar.close)` (¬ß8.1). `current_price` is always `bar.close` ‚Äî deterministic, so live and replay traverse the identical routing path. Routing is gated on handler presence **and** the RUNNING state (PAUSED suspends it without going blind, ¬ß3.1); the no-handler path (replay/inert/test) collects only, and `signals_pulled` still counts unconditionally (Phase D preserved). `_tick` is untouched, so watchdog `record_bar`/heartbeat ordering is byte-for-byte unchanged. ADR-006: the driver is the sole runtime caller of `process_signal`; the `SignalSource` seam holds no handler handle (statically guarded). The Phase E/F "routes nothing" AST guard inverted to `test_driver_routes_through_process_signal`; every watchdog behavioral test is unchanged. Deliberately narrow ‚Äî no orders/fills/sizing/risk/telemetry. Added `tests/runtime/test_driver_execution_routing.py` (11 tests) + a `process_signal` recording spy on `FakeExecutionHandler`; runtime suite **155 passing**. Runtime flow now `Bar ‚Üí Clock ‚Üí SignalSource ‚Üí LoopDriver ‚Üí ExecutionHandler ‚Üí RuntimeWatchdog ‚Üí RuntimeEventJournal`. **Deferred (close Phase G next):** per-signal exception isolation (¬ß8.4, `BROKER_ERROR`) and the IN-001 `KILL_SWITCH_ACTIVATED` single-source migration ‚Äî now reachable, since `process_signal` can trip the handler's own kill switch while the driver still observes only `watchdog.data_healthy`.
*Ref: docs/DRIVER_SPECIFICATION.md ¬ß8; ADR-005, ADR-006; ARCHITECTURE_DECISIONS.md IN-001; docs/PROJECT_STATE.md.*

## 2026-06-06 ‚Äî LoopDriver Startup Gate / Recovery (Phase F)
Added the ¬ß11 startup-validation gate to the `LoopDriver` (`core/runtime/driver.py`), run inside `run()` before the loop when an `ExecutionHandler` is injected. Sequence: `RECOVERY_STARTED` ‚Üí reuse the handler's already-run recovery (`load_db_state=True`; the driver **never** calls `_replay_state()` ‚Äî ADR-001) ‚Üí `RECOVERY_COMPLETED` ‚Üí reconcile the restored ledger against broker truth (`handler.reconciliation.reconcile(broker_positions)`) ‚Üí empty = `RECONCILIATION_PASS` ‚Üí `RUNNING`, non-empty = `RECONCILIATION_FAIL` ‚Üí `abort_startup()` (STOPPED + critical alert; tick loop never runs). Reconciliation branches by **source presence** (require flag + injected `broker_positions`), not by mode; vacuously clear for paper/replay and for LIVE until the broker book is wired (Planned #6). **LIVE mode now requires an injected `ExecutionHandler`** ‚Äî `run()` raises `RuntimeError` without one; the no-handler ungated path is replay/inert/test only. Still execution-free ‚Äî no `process_signal` (ADR-006); the watchdog guard was narrowed from "no `ExecutionHandler` dependency" to the real ADR-006 invariant (no `process_signal` call), allowing the legitimate handler reference the gate requires. Phase E watchdog tests updated to inject a `FakeExecutionHandler`; added `FakeExecutionHandler`/`FakeReconciliation` + `test_driver_startup_gate.py`; runtime suite increased to **144 passing**.
*Ref: docs/DRIVER_SPECIFICATION.md ¬ß11; ADR-001, ADR-006; docs/PHASE_F_STARTUP_GATE_PLAN.md; docs/PROJECT_STATE.md.*
