**Status:** IMPLEMENTED — architecture approved and implemented; see `docs/reports/MM12_5_IMPLEMENTATION_REPORT.md`

---

## 0. Mission

Define the lifecycle of a strategy from creation through live deployment. Every future
strategy must follow this pipeline before it can trade live. The pipeline answers three
questions:

1. **How does a strategy prove it is safe?** By producing mandatory, mechanical evidence
   at each stage — conformance reports, replay proofs, journal audits, reconciliation
   captures — generated by the certified platform's existing instruments.
2. **How does it earn promotion?** By a recorded decision of the named granting
   authority, against a fixed evidence checklist, filed as a permanent ledger entry.
3. **How can promotion be revoked?** Automatically, on any of a fixed set of runtime
   triggers (§7.6), or at operator discretion — always downward, always ledgered, never
   silently.

### 0.1 What this pipeline consumes (and must never modify)

The pipeline is a governance layer over certified Platform Infrastructure v1.0 and the
MM12 strategy-integration stack. It **consumes** these systems as evidence-producing
instruments; it does not redesign, extend, or modify any of them:

| System | Role in the pipeline |
|---|---|
| `SignalSource` ABC (ADR-016) | the interface a candidate implements |
| `SignalSourceConformanceSuite` (MM12.2) | the Stage 1 certification instrument |
| `GuardedSignalSource` (ADR-018/019) | the Stage 2+ runtime enforcement instrument |
| `LoopDriver` + `scripts/fno_runner.py` | the only runtime a candidate is ever validated in |
| `ExecutionHandler` gates (risk, drawdown, stacking, margin, kill switch) | the risk instruments PAPER/LIVE evidence is measured against |
| `NseMarginEngine` / `MarginCalculator` (ADR-011/013) | the sizing authority whose behavior Stage 2/3 evidence must exercise |
| Runtime event journal + `RuntimeMetric` telemetry | the evidence recorders |
| Persistence (SQLite ledger, DuckDB audit projection) | the trade-truth evidence source |
| Startup reconciliation gate + token re-key (ADR-MM7F-1, ADR-MM7J3) | the Stage 3/4 broker-truth instrument |
| Replay (recorded-corpus runs through the real root) | the determinism instrument at every stage |

A promotion requirement that would need a change to any component above is, by
definition, mis-specified. If real evidence ever shows the instruments are insufficient,
that is a platform milestone with its own ADR — not a promotion-pipeline edit.

## 1. Governing principles

1. **The platform certifies safety, not alpha.** Promotion attests that a strategy is
   contract-obedient, deterministic, observable, and risk-bounded — never that it is
   profitable. Profitability is the Account Owner's go/no-go input at Stage 3→4, using
   the PnL facts the evidence dossier must contain, but it is not a pipeline criterion
   (ADR-005: the platform is explicitly not responsible for generating alpha).
2. **Evidence is mechanical or it is not evidence.** Every mandatory item is the output
   of an existing platform instrument (conformance suite, journal, telemetry archive,
   ledger query, replay diff). Narrative claims ("we believe it is deterministic") carry
   zero weight.
3. **Promotion is a revocable license, not a diploma.** Stage 4 carries standing
   obligations and automatic revocation triggers. The ladder has no terminal safe state
   while capital is at risk.
4. **One-way identity.** A certification binds to an exact program (§2). Any change to
   code, config, or contract version produces a new identity that starts at
   Stage 0. There is no "minor patch" fast path — re-certification is cheap by design
   (MM12.1 §14.1); skipping it is how silent regressions reach capital.
5. **Downgrades never require permission; upgrades always do.** Any party (or automatic
   trigger) can demote/suspend instantly. Only the granting authority can promote.
6. **Everything is ledgered.** Every transition — up, down, or sideways — is an
   append-only entry in the Strategy Promotion Ledger with links to immutable, committed
   evidence (§9). An unledgered promotion does not exist.

## 2. Certification identity

A promotion attaches to a **certification identity**, the triple:

```
(strategy_id, code_ref, config_hash)  @ STRATEGY_CONTRACT_VERSION
```

- **`strategy_id`** — the identifier carried on every emitted signal's metadata and in
  every journal record (e.g. `reference_heartbeat_v1`). Any logic change requires a new
  `strategy_id` (MM12.1 §9, fixed rule).
- **`code_ref`** — the strategy repository's commit hash (strategies are external
  packages per ADR-016; the platform records the ref, it does not host the code).
- **`config_hash`** — hash of the exact `build_signal_source(config)` dict used for
  certification. A config *value* change within the certified schema (e.g. a threshold)
  is a new identity; there are no "tunable in production" parameters. A config *schema*
  change is a code change.

The identity is recorded in every ledger entry and every evidence document. Evidence
produced under one identity is void for any other. `STRATEGY_CONTRACT_VERSION` (currently
`"1.0"`) is recorded alongside; a major contract version bump voids all standing
certifications (MM12.1 §14.3) — every strategy re-enters at Stage 1.

## 3. The promotion ladder

Five stages. Exactly the mandated set — evaluation found no evidence justifying
additional *stages* (§3.2). Two **non-ladder states** (SUSPENDED, RETIRED) exist as
revocation machinery, not promotion rungs.

```
Stage 0          Stage 1        Stage 2            Stage 3           Stage 4
DEVELOPMENT ──► CONFORMANT ──► PAPER VALIDATED ──► LIVE CANDIDATE ──► LIVE APPROVED
(external repo,  (passes         (N sessions +      (go-live          (funded, capped,
 no platform     conformance     M trades through   checklist:        under standing
 claims)         Layers 1+2;     the real root in   everything        surveillance;
                 evidence        ExecutionMode.     provable          revocable at
                 dossier         PAPER; zero guard  without capital   any time)
                 opened)         events; replay +   is proven; ops
                                 journal audit      + reconciliation
                                 clean)             + margin capture)
```

### 3.1 State diagram

```
                      ┌────────────────────────────────────────────────────────┐
                      │  ANY change to (strategy_id, code_ref, config_hash)    │
                      │  or a major STRATEGY_CONTRACT_VERSION bump             │
                      │  ⇒ new identity ⇒ re-enter at Stage 0                  │
                      └────────────────────────────────────────────────────────┘

  ┌─────────────┐  P1   ┌─────────────┐  P2   ┌───────────────┐  P3   ┌──────────────┐  P4   ┌──────────────┐
  │  Stage 0    │──────►│  Stage 1    │──────►│   Stage 2     │──────►│   Stage 3    │──────►│   Stage 4    │
  │ DEVELOPMENT │       │ CONFORMANT  │       │PAPER VALIDATED│       │LIVE CANDIDATE│       │LIVE APPROVED │
  └─────────────┘       └─────────────┘       └───────────────┘       └──────────────┘       └──────────────┘
        ▲   ▲  F1: any conformance │   F2: any guard event,  │  F3: checklist item │   T-auto / T-ops:
        │   │  failure             │   DD breach, replay     │  unmeetable, or     │   automatic trigger or
        │   └──────────────────────┘   mismatch, dirty audit │  evidence expiry    │   operator discretion
        │                          └─────────────────────────┘  (re-run Stage 2    │
        │                                                       re-confirmation)   ▼
        │                                                                   ┌─────────────┐
        │    resume (cause external to strategy; grantor sign-off) ◄────────│  SUSPENDED  │
        │                                                                   └─────────────┘
        │    cause is the strategy ⇒ fix ⇒ new identity ⇒ Stage 0                  │
        └───────────────────────────────────────────────────────────────────────────┘
                                                                                    │ decommission
              ┌─────────┐                                                           ▼
              │ RETIRED │◄────────── voluntary, from any stage ────────────── (terminal; strategy_id
              └─────────┘                                                      never reused)
```

- **P1–P4** — promotions: granted only by the named authority (§8), only against the
  complete evidence checklist, always ledgered.
- **F1–F3** — validation failures: automatic demotion to the stage shown; a failure
  caused by the strategy program itself requires a fix, hence a new identity, hence
  Stage 0.
- **SUSPENDED** — reachable from Stage 4 (and Stage 2/3 runs) instantly, by automatic
  trigger or operator discretion. Resume without ladder restart is possible **only**
  when the recorded root cause is external to the strategy (broker outage, platform
  defect, ops error) and the grantor signs off. Any strategy-attributable cause exits
  through Stage 0.
- **RETIRED** — terminal. A retired `strategy_id` is never reused (journal history must
  stay unambiguous).

### 3.2 Why exactly five stages

Additional stages were considered and rejected for lack of evidence:

- **A separate "BACKTESTED" stage before CONFORMANT** — rejected. The external backtest
  is strategy-owned evidence produced in Stage 0 (ADR-016: strategies are external;
  ADR-002: the platform hosts no research). Making it a platform stage would put the
  platform in the business of validating alpha research, violating §1.1.
- **A "LIVE RAMP" stage between 3 and 4** — rejected as a stage; the same control exists
  as a Stage 4 standing obligation (initial capital cap + ledgered cap raises, §7.5).
  A ramp is a parameter of LIVE APPROVED, not a distinct evidence gate — it produces no
  new *kind* of evidence, only more of the Stage 4 kind.
- **A "RE-CERTIFICATION" stage for changed strategies** — rejected; MM12.1 §14.1 already
  fixed the rule that a changed strategy is a new identity entering at Stage 0/1. A
  dedicated stage would create pressure for a shortcut path, which is exactly what the
  fixed rule exists to prevent.

The one structural addition relative to MM12.1's three-gate sketch — splitting
LIVE CANDIDATE out of LIVE-APPROVED — is not a new invention: it executes MM12.1 §15's
exit criterion for MM12.5 verbatim ("checklist explicitly separates architectural
completion from the funded-account operational prerequisite"). Stage 3 is the
architectural-completion half; Stage 4 is the funded-operational half.

## 4. Stage definitions

Each stage defines: purpose, entry criteria, required evidence, required tests, required
documentation, required replay evidence, exit criteria, failure criteria, rollback
criteria.

### 4.0 Stage 0 — DEVELOPMENT

| Field | Definition |
|---|---|
| **Purpose** | The strategy exists as an external repository/package (ADR-016). The platform makes no claims about it and grants it nothing. All research, backtesting, and iteration happen here, outside the platform (ADR-002). |
| **Entry criteria** | None enforced. Practically: a `strategy_id` is reserved in the Promotion Ledger (one line, status DEVELOPMENT) so identity collisions are impossible. |
| **Required evidence** | None *held* by the platform. The stage's job is to *produce* the Stage 1 submission package (below). |
| **Required tests** | Strategy-owned. The platform does not run or grade them. |
| **Required documentation** | None yet; drafted here for Stage 1: the Strategy Datasheet (§9.2). |
| **Replay evidence** | None required by the platform. The strategy owner is on notice that Stage 1 will mechanically verify replay-equivalence — a strategy not built deterministic (MM12.1 §6 forbidden list) will fail there. |
| **Exit criteria (→ Stage 1 submission)** | A complete submission package: (a) strategy package with `build_signal_source(config)` factory export; (b) pinned `code_ref` + certification config; (c) Strategy Datasheet including the **risk declaration** (§7.4.1) and declared `on_bar` latency budget; (d) the external backtest report (strategy-owned evidence of intent — filed, not graded); (e) target `STRATEGY_CONTRACT_VERSION`. |
| **Failure criteria** | N/A — nothing granted, nothing to fail. |
| **Rollback criteria** | N/A. Abandonment = mark the reserved `strategy_id` RETIRED. |

### 4.1 Stage 1 — CONFORMANT

| Field | Definition |
|---|---|
| **Purpose** | Prove **contract obedience**: the candidate is a well-formed, deterministic, side-effect-free `SignalSource` that emits contract-clean signals. This is the gate that earns the guard's assumption of contract-cleanliness (MM12.1 fixed rule #1: no strategy reaches PAPER without CONFORMANT). |
| **Entry criteria** | Complete Stage 0 submission package; identity triple pinned. |
| **Required evidence** | (1) **Conformance report**: full `SignalSourceConformanceSuite` output — every Layer 1 static check and Layer 2 behavioral check PASS, over a named, committed, recorded bar corpus (corpus identity recorded in the report). (2) **Guard-wrap conformance**: `GuardedSignalSource(candidate)` also passes the full suite (the MM12.4 precedent, now mandatory — proves the strategy composes with the runtime enforcement layer). (3) The risk declaration cross-checked for internal consistency (declared universe matches corpus symbols; `sl_distance`/`risk_r` semantics stated). |
| **Required tests** | The conformance suite **is** the test. Additionally: the strategy package's own test suite green at `code_ref` (recorded, not graded — the platform grades only conformance). |
| **Required documentation** | Strategy Datasheet v1 (frozen at grant): identity triple, contract version, config schema + certified config values, universe, session behavior, latency budget, risk declaration, external backtest reference. |
| **Replay evidence** | The suite's replay-twice determinism check: two fresh instances over the identical corpus emit byte-identical signal streams (MM12.1 §6, §7.2). This is the *offline* determinism proof; Stage 2 adds the end-to-end one. |
| **Exit criteria (→ Stage 2)** | Grantor (§8) reviews the evidence dossier, grants CONFORMANT, and files the ledger entry. Grant recorded with the corpus identity and suite version. |
| **Failure criteria** | Any suite check fails; datasheet inconsistent with observed behavior; corpus unrepresentative of declared universe. Failure → remains DEVELOPMENT; a code fix → new identity. |
| **Rollback criteria** | CONFORMANT is revoked automatically by: identity change (always), a major contract-version bump, or discovery that the certification corpus/evidence was defective (ledgered as an evidence-defect entry; all downstream grants void simultaneously). |

### 4.2 Stage 2 — PAPER VALIDATED

| Field | Definition |
|---|---|
| **Purpose** | Prove behavior **through the real composition root against real market data with zero capital**: `scripts/fno_runner.py`, `ExecutionMode.PAPER`, wrapped unconditionally in `GuardedSignalSource` (ADR-020 execution). Conformance proved the strategy *can* behave; PAPER proves it *does* behave — across real sessions, real data gaps, real restarts, and the full data→guard→driver→execution→risk→margin→PaperBroker→ledger→telemetry→journal pipeline (the MM12.4-proven path). |
| **Entry criteria** | CONFORMANT granted; PAPER deployment configured with the certified config; handler risk gates configured from the risk declaration (drawdown gate = declared max DD; daily trade limit = declared frequency bound). |
| **Required evidence** | (1) **Session evidence**: ≥ the minimum window (§7.3 — 20 trading sessions AND 30 completed round-trips). (2) **Guard cleanliness**: `STRATEGY_ERRORS = 0`, `STRATEGY_QUARANTINE_EVENTS = 0`, `SIGNAL_CONTRACT_REJECTIONS = 0` across the entire window — one guard event fails the stage (§7.6; a certified strategy has no excuse at the boundary). (3) **Journal audit** (ADR-017 consequence, mandated for this gate): emitted intent vs. executed reality — every emitted signal traced to fill / journaled gate-rejection; shadow-state divergence quantified and explained (§7.4.4). (4) **Risk metrics report** from the ledger (§7.4.2), including max DD vs. declared max DD. (5) **Telemetry archive**: per-session `RuntimeMetric` snapshots + heartbeat continuity (§7.2). (6) **Margin evidence**: every entry passed the margin gate with `NseMarginEngine`-computed margin journaled; an F&O strategy must show SPAN/ELM figures exercised, closing the MM12.4 named limitation (§7.7). (7) **Kill-switch drill**: one operator drill during the window (§7.8). |
| **Required tests** | None new — the running platform is the test. The full platform regression suite must be green at the platform commit the PAPER window ran on (recorded in the report). |
| **Required documentation** | **PAPER Validation Report** (permanent record, §9.2): window dates, platform commit, identity triple, all seven evidence items, restarts and their causes, anomalies and dispositions. |
| **Replay evidence** | End-to-end: ≥1 full session's recorded bar corpus re-driven through the real composition root; the emitted signal stream is byte-identical and the ledger's deterministic fields (symbol, side, quantity, fill price, signal_id) match the original session's, with the standing exclusions (`broker_id` UUIDs, journal wall-clock timestamps — MM12.4 §7.2). PaperBroker fills are deterministic, so this diff is exact, not statistical. |
| **Exit criteria (→ Stage 3)** | Grantor reviews the PAPER Validation Report; grants PAPER VALIDATED; ledger entry. |
| **Failure criteria** | Any guard event; declared-max-DD breach (the handler's drawdown gate tripping *is* the breach signal); replay mismatch; journal audit shows unexplained divergence; a strategy-attributable process incident. Failure → back to CONFORMANT if no code change is needed (e.g. corpus/config error on the platform side), else → Stage 0 via new identity. A strategy-caused restart resets the session window clock. |
| **Rollback criteria** | Automatic on identity change or contract bump (as Stage 1). Additionally **evidence expiry**: if more than 60 calendar days pass between the PAPER window's end and a Stage 3→4 grant, a 5-session PAPER re-confirmation (same evidence kinds, abbreviated window, zero guard events) is required — validation evidence decays as market regime and platform commits move on. |

### 4.3 Stage 3 — LIVE CANDIDATE

| Field | Definition |
|---|---|
| **Purpose** | **Architectural completion, separated from capital.** Everything provable without a funded account is proven here, so that the Stage 4 decision is purely the Account Owner's capital decision against a complete dossier — never "approve and we'll finish wiring later." This stage is the MM12.1 §15 exit-criterion split made structural. |
| **Entry criteria** | PAPER VALIDATED granted and unexpired. |
| **Required evidence — the Go-Live Checklist (all items, no partial credit)** | **(a) Broker reconciliation proven live** (LIVE-only requirement, ADR-013/MM14): a first-hand authenticated, non-empty `short-term-positions` capture through the real adapter; the `get_positions() → rekey_broker_positions_by_token() → to_reconcile_positions()` chain shown reconciling the real broker book against the ledger (the standing operational prerequisite from PROJECT_STATE Planned #6, now a named gate). **(b) Broker margin reconciliation report** (ADR-013's deferred LIVE-only capability, built under MM14): broker-quoted margin fetched, compared, and logged against `NseMarginEngine` for representative candidate positions; divergence characterized (parity is structurally unreachable — the report bounds the gap, it does not chase zero). **(c) Credential/token lifecycle** documented and exercised: token issue → expiry → renewal path walked once for real; expiry mid-session behavior confirmed to hit the ADR-MM7F-1 refusal or the watchdog path, never a silent state. **(d) Process supervision**: restart policy defined; heartbeat monitored by something *outside* the process; the full alert path (journal → alerter → operator's device) test-fired and receipt confirmed. **(e) Kill-switch runbook** verified (§7.8): every activation path (manual, watchdog staleness, drawdown, daily-limit) demonstrated on the live-configured deployment running in PAPER mode. **(f) Rollback runbook** (§7.9) written and walked through on the PAPER deployment. **(g) Capital plan**: allocation, initial cap (§7.5), and handler risk-gate settings (drawdown, daily trade limit, margin budget) that match the risk declaration. |
| **Required tests** | Platform regression suite green at the candidate deployment commit. No new strategy tests — the strategy is unchanged from Stage 2 by identity. |
| **Required documentation** | The completed, dated Go-Live Checklist (permanent record); the two runbooks (kill-switch, rollback); the capital plan. |
| **Replay evidence** | None new. Stage 2's replay evidence stands (the strategy is byte-identical by identity; the deployment differences are operational, not behavioral). |
| **Exit criteria (→ Stage 4)** | Checklist 100% complete; **dual sign-off** — Technical Lead (evidence is complete and genuine) **and** Account Owner (capital commitment) (§8); ledger entry records the grant, effective date, initial capital cap, and risk-gate settings. |
| **Failure criteria** | Any checklist item unmeetable (e.g. broker capture cannot be produced) — the candidacy stalls; it does not decay into a partial approval. PAPER-evidence expiry (§4.2) forces the 5-session re-confirmation before grant. |
| **Rollback criteria** | Identity change / contract bump (automatic, as always). Voluntary withdrawal → back to PAPER VALIDATED (the evidence survives; the candidacy is what lapses). |

### 4.4 Stage 4 — LIVE APPROVED

| Field | Definition |
|---|---|
| **Purpose** | Trade live under a **revocable license with standing obligations**. The pipeline does not end at go-live; Stage 4 is an operating regime, not a certificate on the wall. |
| **Entry criteria** | Stage 3 dual sign-off; funded account; risk gates configured exactly as ledgered; `ExecutionMode.LIVE` composition per the frozen root (LIVE reconciliation callable wired — `_make_live_broker_positions`). |
| **Standing obligations (continuous evidence)** | (1) **Startup reconciliation every session** — structural (the gate refuses to start otherwise, ADR-MM7F-1); divergence alerts investigated same-day and ledgered if strategy-attributable. (2) **Telemetry surveillance**: heartbeat + `RuntimeMetric` counters monitored per §7.2; the guard counters are expected to be *permanently zero* in LIVE. (3) **Journal audit cadence**: weekly intent-vs-reality audit (Stage 2 methodology) for the first 4 live weeks, monthly thereafter. (4) **Broker margin comparison logging** (per §7.7's LIVE obligation) at the cadence set in the Stage 3 report. (5) **Capital ramp**: initial cap held for ≥10 live sessions; every cap raise is a ledgered decision by the Account Owner referencing the accumulated live evidence — never a config-drift. |
| **Required tests** | Platform regression green at every deployment commit change (a platform upgrade under a live strategy is an ops event: stop, upgrade, verify suite, restart — the identity is unaffected because the *strategy* is unchanged). |
| **Required documentation** | Ledger entries for: grant, every cap change, every incident, every audit with findings. |
| **Replay evidence** | On demand: any live anomaly must be reproducible by replaying the session's recorded bars through the same identity — Audit-First made operational (§7.1). Inability to replay-explain a live trade is itself a revocation-grade finding. |
| **Exit criteria** | None upward — Stage 4 is the top. Exits are RETIRED (voluntary decommission: flatten, stop, ledger) or SUSPENDED. |
| **Failure / revocation criteria (automatic → SUSPENDED)** | Fixed trigger list, §7.6: any `STRATEGY_QUARANTINED` in LIVE; any `SIGNAL_CONTRACT_REJECTED` in LIVE; kill switch tripped by a strategy-attributable cause; declared-max-DD breach; reconciliation divergence attributable to strategy behavior; a live trade that cannot be replay-explained. Plus operator discretion, always, no justification required in the moment (justification is recorded afterward in the suspension entry). |
| **Rollback criteria** | The rollback runbook (§7.9). Resume from SUSPENDED per §3.1: external cause + grantor sign-off resumes at Stage 4; strategy-attributable cause exits via new identity to Stage 0. |

## 5. Non-ladder states

### 5.1 SUSPENDED

Trading halted, license in abeyance, cause under investigation. Mechanics: kill switch
and/or quarantine already halted signal flow (the platform's *kill-switched-but-running*
idiom — observability survives, ADR-019); the operator manages open positions manually
via the broker (the platform never auto-flattens — ADR-019's rejected alternative stays
rejected). Entry is instant and permissionless (§1.5). Every suspension gets a ledger
entry: trigger, timestamp, open-position disposition, and — within 5 trading days — an
investigation report classifying the root cause as *strategy-attributable* or *external*.
That classification is the fork in §3.1.

### 5.2 RETIRED

Terminal, voluntary, from any stage. Positions flat, process stopped, ledger entry
closed with a retirement note. The `strategy_id` is never reused; the dossier and all
journal history remain permanently in the repository. (Precedent: ADR-020's reference
strategy is effectively born-retired from the ladder — permanently PAPER-confined,
never promotable; the ledger will carry it as a standing exhibit, not a candidate.)

## 6. Movement rules (summary)

1. **Promotions (P1–P4)**: only by the granting authority (§8), only with the stage's
   complete evidence, always a ledger entry. No stage may be skipped, ever — including
   for ports of historical designs (MM12.1 §14.2: architecturally identical to brand-new
   strategies).
2. **Demotions/failures (F1–F3, T-auto, T-ops)**: automatic or discretionary, effective
   immediately, ledgered after the fact if necessary. Safety never waits for paperwork;
   paperwork always catches up (within 5 trading days).
3. **Identity changes**: any change to the triple (§2) is not a "demotion" — it is a
   *new strategy* at Stage 0. The old identity's ledger line is closed with a pointer to
   its successor.
4. **Contract-version bumps**: a major `STRATEGY_CONTRACT_VERSION` change voids every
   standing grant at once; each strategy re-enters at Stage 1 (conformance re-run under
   the new suite). Additive minor changes void nothing (MM12.1 §14.3).
5. **Platform changes**: platform commits do not touch strategy identity, but Stage 2+
   evidence records the platform commit it was produced on, and Stage 4 requires a green
   regression suite before any live deployment moves to a new platform commit.

## 7. Cross-cutting requirements

### 7.1 Replay requirements

| Stage | Replay proof required |
|---|---|
| 1 | Conformance replay-twice: identical corpus ⇒ byte-identical signal streams (offline). |
| 2 | End-to-end: ≥1 full recorded session re-driven through the real root ⇒ identical signal stream + identical deterministic ledger fields (exclusions: `broker_id` UUIDs, journal wall-clock — MM12.4 §7.2). |
| 3 | Stage 2 evidence stands (identity unchanged). |
| 4 | On demand: every live anomaly replay-explainable from recorded bars + journal. Failure to replay-explain is a revocation trigger. |

Recorded bar corpora used as evidence are committed (or content-hashed and archived)
so replay evidence is reproducible years later — evidence that cannot be re-run is
narrative (§1.2).

### 7.2 Determinism & telemetry expectations

- Determinism is the MM12.1 §6 contract, verified mechanically at Stage 1 and
  end-to-end at Stage 2. Nothing new is defined here; the pipeline only *demands* the
  existing proofs.
- Telemetry: every validation window archives per-session `RuntimeMetric` snapshots.
  Mandatory expectations: `BARS_PROCESSED`/`LOOP_ITERATIONS` consistent with the session;
  `SIGNALS_RECEIVED`/`SIGNALS_ROUTED`/`EXECUTION_CALLS` mutually consistent;
  `STRATEGY_ERRORS = STRATEGY_QUARANTINE_EVENTS = SIGNAL_CONTRACT_REJECTIONS = 0` in any
  window offered as promotion evidence; heartbeats present and gap-free in live-data
  modes (Constitution §6 — observable without broker login). A telemetry gap is missing
  evidence: the affected session does not count toward the window.

### 7.3 PAPER duration and minimum trade count

- **≥ 20 NSE trading sessions** (≈ one calendar month: spans at least four weekly
  expiries and one monthly expiry cycle — the platform's real periodicities) **AND**
- **≥ 30 completed round-trips** (entry + exit, ledger-confirmed). 30 is a floor for
  behavioral coverage of the entry/exit/gate surface, not a claim of statistical
  significance — the platform certifies safety, not alpha (§1.1).
- Both must hold; the window extends until they do. If a low-frequency strategy cannot
  produce 30 round-trips in 60 sessions, the grantor may accept the 60-session window
  with the shortfall explicitly ledgered as an accepted deviation (visible forever, not
  waived silently).
- Restart handling: platform/ops restarts don't reset the window (recovery is a
  certified feature); a strategy-attributable incident resets it (§4.2).

### 7.4 Risk metrics, drawdown limits, audit

#### 7.4.1 Risk declaration (authored at Stage 0, frozen at Stage 1)
Declares: max drawdown (Rs and % of allocated capital), expected trade frequency band,
max simultaneous positions, universe, per-trade risk (`risk_r` semantics), max margin
utilization, and `on_bar` latency budget. The declaration configures the handler's
gates for every validation window — the strategy is validated *against its own promises*.

#### 7.4.2 Risk metrics report (Stage 2 evidence; Stage 4 audit cadence)
Computed from the ledger (single source of truth, ADR-001): round-trip count, win rate,
average win/loss in R, profit factor, max drawdown (Rs, %), peak gross exposure, peak
margin utilization, signal→fill conversion with per-gate rejection breakdown, guard
counters. PnL facts are included for the owner's judgment but are not pass/fail (§1.1).

#### 7.4.3 Drawdown limits
The declared max DD is the binding limit at every stage that trades (2 and 4). The
handler's existing drawdown gate is set to it; the gate tripping **is** the breach
event — Stage 2: validation failure; Stage 4: automatic suspension. There is no
"explainable breach" carve-out: a strategy that breaches its own declaration has either
a defect or a wrong declaration, and both require a new identity.

#### 7.4.4 Audit requirements
Audit-First (Constitution/CLAUDE.md Principle 5) operationalized: every trade in every
evidence window traceable bar → signal (with strategy metadata) → order → fill → ledger
row, no missing link. The Stage 2 journal audit and Stage 4 audit cadence verify this
chain and quantify shadow-state divergence (ADR-017's mandated PAPER-gate item):
divergence is expected to be one-directional (strategy believes in rejected entries) —
any *reverse* divergence is a platform defect and halts the pipeline for everyone, not
just the candidate.

### 7.5 Capital controls (Stage 4)

Initial cap: the minimum viable size for the instrument (one lot for F&O; the risk
declaration's minimum for equity), held ≥10 live sessions. Raises: Account Owner
decision, ledgered, referencing accumulated live evidence. No automatic scaling.

### 7.6 Guard events and automatic revocation triggers

Guard events are **binary-quality signals** by construction: a CONFORMANT strategy has
mechanically proven it never emits contract-violating signals and never throws on the
certification corpus. Therefore in any post-certification window:

| Event | In PAPER window | In LIVE |
|---|---|---|
| `STRATEGY_ERROR` / `STRATEGY_QUARANTINED` | Stage 2 failure | automatic suspension |
| `SIGNAL_CONTRACT_REJECTED` (even one) | Stage 2 failure | automatic suspension |
| Kill switch, strategy-attributable cause | Stage 2 failure | automatic suspension |
| Declared-max-DD breach (drawdown gate trip) | Stage 2 failure | automatic suspension |
| Reconciliation divergence, strategy-attributable | n/a (PAPER book is empty) | automatic suspension |
| Live trade not replay-explainable | n/a | automatic suspension |

This resolves MM12.1 open question #3 (escalating contract-violation counts to
quarantine) *for promotion purposes* without touching the guard: the guard's v1.0
runtime policy (drop-and-journal, no escalation) is unchanged; the *pipeline* treats
the first journaled rejection as certification-voiding. No code change — the journal
already records the evidence.

### 7.7 Margin validation

- **Stages 1–2:** all margin figures are `NseMarginEngine`/`MarginTracker` outputs — the
  sole sizing authority in every mode (ADR-011/013; unchanged PAPER vs LIVE by design).
  Stage 2 evidence must show the margin gate exercised on every entry; an F&O strategy's
  PAPER report must show real SPAN + ELM numbers flowing (closing the MM12.4 named
  limitation for the first F&O candidate).
- **Stage 3:** the broker margin reconciliation report (checklist item b) — the ADR-013
  deferred capability, built under MM14 when the first candidate reaches this stage,
  never before (no abstraction ahead of need).
- **Stage 4:** ongoing broker-vs-local margin comparison logging at the ledgered cadence.
  The broker RMS remains order-acceptance authority only; it never overrides sizing
  (ADR-013 — the pipeline enforces the two-authorities model, it does not revisit it).

### 7.8 Kill-switch expectations

| Stage | Expectation |
|---|---|
| 2 | One operator drill inside the window: activate the kill switch, observe entry signals blocked and journaled, observe the *kill-switched-but-running* posture (telemetry/heartbeat alive), restart per runbook draft. |
| 3 | Every activation path demonstrated on the live-configured deployment in PAPER mode: manual, watchdog staleness trip, drawdown gate, daily-trade-limit. Runbook finalized. |
| 4 | Standing: any trip is investigated; strategy-attributable trips suspend automatically (§7.6). Kill-switch drills repeat after any ops-stack change (supervision, credentials, alerting). |

### 7.9 Rollback procedures (the runbook, fixed order)

1. **Halt intake** — activate the kill switch (blocks everything including EXITs —
   known posture) and/or confirm quarantine already latched.
2. **Assess positions** — read the ledger (truth, ADR-001), cross-check the broker book.
3. **Disposition positions** — operator decision, manually via broker if flattening;
   never automated (ADR-019).
4. **Stop the process** — clean shutdown; journal shows STOPPED.
5. **Ledger the demotion/suspension** — trigger, timestamp, position disposition.
6. **Investigate** — report within 5 trading days; classify cause (§5.1).
7. **Re-entry** — only through §3.1's two paths. Never restart the process "to see if
   it recurs": a deterministic strategy that faulted will fault again (ADR-019), and a
   nondeterministic one has already voided its certification.

## 8. Authority model — who grants promotion

Two named roles. Today one person holds both hats; the roles are still recorded
separately on every grant because they answer different questions, and the dual-key
structure is what makes a future second person a ledger change, not a governance
redesign.

| Role | Authority | Grants |
|---|---|---|
| **Technical Lead** (platform owner) | "Is the evidence complete, mechanical, and genuine?" | P1 (CONFORMANT), P2 (PAPER VALIDATED), P3 (LIVE CANDIDATE); the technical half of P4; resume-from-SUSPENDED sign-off |
| **Account Owner** (capital authority) | "Is my capital committed to this?" | The capital half of P4 (LIVE APPROVED); every cap raise; retirement of a live strategy |

Anyone — either role, or an automatic trigger — can demote or suspend (§1.5). The
strategy author *proposes* with evidence and never grants anything, including when the
author is the same person: the grant entry is written against the checklist, not against
confidence.

## 9. Permanent records and audit of promotion decisions

### 9.1 The Strategy Promotion Ledger — `docs/STRATEGY_PROMOTION_LEDGER.md`

Append-only, repository-committed (precedent: `MM11_REMOVAL_LEDGER.md`; commit-hash
pinning precedent: the MM11.7 certification). One entry per transition:

```
date · strategy_id · (code_ref, config_hash) · contract version · from-state → to-state
· evidence links (committed paths) · platform commit · grantor role(s) · note
```

Rules: entries are never edited or deleted (supersede with a correcting entry);
promotions without an entry do not exist (§1.6); the ledger is the index, never the
evidence body (evidence lives in the dossier).

### 9.2 The strategy dossier — `docs/strategies/<strategy_id>/`

The permanent evidence corpus per identity:

| Document | Created at | Permanent record because |
|---|---|---|
| Strategy Datasheet (incl. risk declaration) | Stage 0→1 | the promises every window is validated against |
| Conformance Report | Stage 1 | the certification basis |
| External backtest report (filed, not graded) | Stage 1 | the owner's intent evidence, referenced at Stage 3→4 |
| PAPER Validation Report | Stage 2 | the zero-capital behavioral proof |
| Go-Live Checklist (completed, dated) | Stage 3 | the architectural-completion proof |
| Kill-switch + rollback runbooks | Stage 3 | the operational contract |
| Grant/suspension/incident/audit entries' evidence | Stage 4 | the live operating record |
| Investigation reports | any | why a demotion happened; the classification that gated re-entry |

### 9.3 How promotion decisions are audited

Every decision is reconstructable from committed artifacts alone: the ledger entry names
the evidence; the evidence names the corpus, the platform commit, and the identity
triple; the corpus and the code are re-runnable (§7.1). Auditing a grant = re-running
its evidence. Auditing the *pipeline* = walking the ledger and confirming every live
session's `strategy_id` (present in every journal record) maps to a then-current
LIVE APPROVED entry — a mechanical scan, performable at any time, with no reliance on
anyone's memory.
