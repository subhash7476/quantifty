## Open Findings

- **F3 — Tick Size Scaling Defect — OPEN DEFECT.** `CanonicalInstrument.tick_size` is stored ≈100× too large: the live Upstox payload encodes `tick_size` in **paise**, and `parse_instruments` does `float(tick_size or 0.0)` with **no `/100`**, so the DB (and the resolver, which carries it through unscaled) holds paise, not rupees — e.g. a NIFTY option stores `5.0` where the canonical rupee tick is `₹0.05`. The whole distinct-value set is clean paise (`1/5/10/50/100/500/1000`). **Current impact: none** — nothing in `core/` consumes `CanonicalInstrument.tick_size` yet (the legacy options path reads expiry/strikes/lot, never tick), and the canonical layer is dormant pre-4C.7. **Future impact: potentially execution-affecting** — once 4C.7 wires canonical attributes into order pricing/rounding, a 100× tick mis-rounds prices. **Owner: parser / 4C.7 preparation slice** (normalize at ingest or scale at the resolver boundary, with a paise→₹ regression test). **Do NOT silently fold into another change — track explicitly.** *(detail: MM.5 report Finding 3)*
- **F4 — Lot Size Discrepancy — INVESTIGATION REQUIRED.** The materialized master carries **NIFTY lot_size = 65** and **BANKNIFTY = 30** (raw from the live Upstox master, correct unit, unscaled). This differs from the prior repo/fixture assumption of **NIFTY = 75** (`test_master_ingest.py`; CLAUDE.md's "post-2024 SEBI 50→75" note). **Neither value is asserted here.** Before any code change, verify the materialized values against the authoritative sources — **exchange circulars, broker master, and current contract specifications** — and treat the live exchange specification as authoritative. **Current impact: none. Future impact: potentially severe** (lot size sizes every F&O order). **Do not "fix" until verified.** *(detail: MM.5 report Finding 4)*
